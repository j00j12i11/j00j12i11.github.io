## 0. 사고 요약

정처기 시험 직전의 새벽. 지금까지 공부한 내용의 총집합이라 할 수 있는 요약본을 정리하기 전에 개념노트의 형식과 깨진 위키링크부터 한 번 손보고 싶었다.

그 가벼운 부탁이 **Edit 212회 / 약 60M 토큰**짜리 사고로 끝났다. Claude는 주어진 임무를 잘 수행했다. 결과물은 멀쩡했고 링크도 다 맞춰졌다. 다만 상상치도 못한 비효율적인 방식으로 그 새벽 공부에 쓰려던 내 토큰을 다 녹여버렸다.

이 글은 왜 그런 일이 발생했는지 transcript를 뒤져 되돌아보고 Claude Code를 쓰는 동안 주의해야 할 것에 경각심을 다지는 기록이다.

## 1. 발단

이런 사고가 나기까지는 다음의 원인들이 복합적으로 작용했다.

- 에이전트 배제
- acceptEdits + 자리 비움
- Claude의 도구 선택 미스

### 1) 에이전트 배제

Claude Code에서 내가 직접 대화하는 메인 세션 이외에, sub-agent를 호출해서 작업하는 것도 가능하다. sub-agent는 자기 컨텍스트를 따로 가지기 때문에, 대량 파일 검사·반복 치환 같은 무거운 작업을 맡겨두면 메인 세션의 컨텍스트를 아낄 수 있다.

처음에는 링크가 깨진 것 같으니 전체적으로 한번 확인해 달라고 했다. Claude는 정석대로 sub-agent를 띄워 리포트를 가져왔다. 231개 중 181개가 깨진 상태였으므로 다시 수정 명령을 보냈다. 모든 회차의 위키링크를 정리하고, 개념노트의 헤딩 형식까지 같이 바꾸는 일괄 작업이었다. 이번에는 sub-agent를 빼고 메인 세션이 직접 처리하도록 했다.

두 가지 이유가 겹친 결정이었다. 하나는 수정 항목을 보니 한 번에 정확히 처리하고 싶어서 더 믿음직한 메인 세션에 맡기고 싶었기 때문이고, 다른 하나는 sub-agent 자체에 대한 신뢰가 이미 떨어져 있었기 때문이다.

> 정처기 학습 사이클(문제 풀이 → 오답 복기 → 개념노트 정리)에 sub-agent를 끼워보려 했지만 잘 안 굴러갔다.
>
> - "5문제마다 자동 호출" 트리거가 안정적으로 발화하지 않았다. CLAUDE.md에 횟수·시점 기준을 적어둬도 빈번히 안 잡혔다.
> - sub-agent를 띄울 때마다 시작·종료 오버헤드가 컸고, 5문제 사이에 끼우기엔 학습 흐름이 매번 끊겼다.
> - 결국 sub-agent에 명령을 주고 결과를 검증하는 일 자체가 기출 복기 사이클을 방해했다.

### 2) acceptEdits + 자리 비움

Claude Code는 기본적으로 사용자의 허가를 받고 나서 동작한다. 그런데 노트들을 구성하면서 발생할 Edit이 한두 개일 리가 없었다. 매번 수락하기는 귀찮으므로 acceptEdits 모드로 풀어둔 상태였다.

그리고 일을 시킨 뒤 잠시 다른 볼일을 보고 있었다. Claude는 권한을 가지고 있고, 중간에 확인할 사람도 없었던 셈이다.

### 3) Claude의 도구 선택 미스

Claude가 도구를 잘못 선택했다. 같은 패턴의 위키링크 치환을 라인별 Edit으로 하나씩 처리한 결과, 212회 중 `replace_all`을 쓴 건 단 한 번이었다. 단일 호출로 끝낼 수 있는 작업이었다.

## 2. 그날 일어난 일

내가 부탁한 일 자체는 단순했다. 개념노트의 H2 헤딩에서 괄호 안 부연 설명을 떼고, 그에 맞춰 모든 회차의 기출복기 파일에 있는 위키링크를 갱신하는 것. 치환 패턴도 깔끔했다.

- `[[정보보안#SSH (Secure Shell)]]` → `[[정보보안#SSH]]`
- `[[네트워크#L2TP (Layer 2 Tunneling Protocol)]]` → `[[네트워크#L2TP]]`
- `[[네트워크#SSL/TLS (Secure Sockets Layer/Transport Layer Security)]]` → `[[네트워크#SSL/TLS]]`

영향을 받는 파일은 총 18개.

- 기출복기 11개: 2020-1회부터 2023-1회까지, 회당 14~20번씩 Edit
- 개념노트 6개: 정보보안 17번, 네트워크 10번, SQL 5번, 데이터베이스 3번 등
- CLAUDE.md 1번

그런데 Claude는 라인별 Edit을 골랐다. 한 파일에 같은 패턴이 여럿 있어도 매 라인을 1:1로 처리했고, 매 Edit 직전에 같은 파일을 다시 읽었다. 한 파일을 20번 다시 열면 그 내용이 20번 컨텍스트에 쌓였다. **내 토큰은 그렇게 사라졌다.**

더 좋은 방법이 있었다. `Edit`의 `replace_all`이면 한 호출로 같은 패턴을 한 파일에서 다 바꿀 수 있고, `MultiEdit`이면 한 호출로 여러 편집을 묶을 수 있다. `Bash + sed`라면 여러 파일에 동시에 적용할 수도 있었다.

결과적으로 cache_creation 토큰이 매 Edit 응답마다 수만에서 수십만씩 누적됐고, 후반부에는 한 호출에 252,851까지 잡혔다.

Claude는 같은 내용이 반복되면 그걸 캐시에 저장해두고 다음번엔 싸게 가져다 쓴다. 처음 저장할 때(cache_creation)는 일반 가격, 두 번째부터 가져다 쓸 때(cache_read)는 1/10 가격이다.

그런데 매 Edit마다 같은 파일을 새로 읽으면 그 파일 내용이 캐시에서 재사용되지 않고 매번 새로 저장된다. 1/10로 가야 할 것이 매번 일반 가격으로 잡히는 셈. cache_read 대신 cache_creation만 계속 쌓였다.

## 4. 가드레일

결과물이 멀쩡했고 링크도 다 맞춰져 있어서, 토큰 사용량을 직접 확인하기 전까지는 일이 잘 끝난 줄 알았다. 다음 작업을 진행하려는데 꽉 찬 세션 사용량을 보고 뭔가 잘못됐다는 걸 알았다. 그제야 직전 세션 로그를 뒤져 상황을 확인했다.

정리하면서 두 가지를 조정했다.

첫째, "에이전트 쓰지 말고 메인 세션이 하라"는 명령을 거뒀다. 에이전트를 한 번 띄워봤는데 결과가 흡족하지 않아서 "그냥 메인 세션이 해"로 돌렸을 뿐이었는데, 그 한 줄이 작업 전체를 메인 세션 단일 컨텍스트에 몰아넣었다.

둘째, Obsidian Vault의 CLAUDE.md에 다음의 가드레일을 작성했다.

<div class="guardrail-callout">

> **대량 편집 가드레일**
>
> Obsidian Vault 는 위키링크·frontmatter·메타블록 등 반복 패턴이 많아 대량 치환 작업이 자주 발생한다. 아래 기준에 도달하면 **반드시 멈추고 전략을 재검토한 뒤 사용자에게 제안**한다.
>
> - 동일 파일 Edit 5회 초과 → Write 전체 재작성 검토
> - 동일 패턴 치환 10회 초과 → Bash + sed/python 스크립트 전환 제안
> - 한 세션에서 Edit 50회 초과 → 즉시 중단하고 일괄 처리 방식 재설계
>
> **이유**: 동일 파일을 반복 Edit 하면 프롬프트 캐시가 매번 무효화되어 토큰이 수십 배로 불어난다. 실제 사례: 위키링크 일괄 치환을 Edit 212회로 수행해 단일 세션에서 59M 토큰 소비.
>
> **예외**: 패턴이 파일마다 다르거나, 맥락 판단이 필요한 수정은 Edit 유지.

</div>

## 5. 마무리하며

이 일은 Claude Code 같은 에이전트와 일할 때 규제와 가드레일이 왜 필요한지 직접 부딪혀 배운 사고였다. 결과물만 보면 일은 끝나 있다. 그런데 그 과정에서 컨텍스트가 어떻게 쌓였고 비용이 얼마나 들었는지는 결과물에 드러나지 않는다. **결과만 보지 말고 과정에도 주의를 기울여야 한다.**

그래서 요즘 컨텍스트 엔지니어링이나 에이전트 하네스 같은 주제에 관심이 더 간다. 컨텍스트를 어떻게 관리하고, 어떤 도구·권한을 어떻게 제약하고, 사람이 개입할 지점을 어디에 둘지. 이번 사고는 거기에 관심이 가게 만든 직접적인 계기였다.
