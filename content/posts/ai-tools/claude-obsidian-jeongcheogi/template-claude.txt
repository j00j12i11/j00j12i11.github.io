# 학습용 볼트 — 운영 규칙

> 이 파일은 Claude(또는 Codex)가 이 볼트에서 작업할 때 매번 참조하는
> 마스터 규칙이다. 도메인·주제별 세부 규칙은 별도 파일로 두고, 이 파일은
> "어떤 트리거에 어떤 규칙 파일을 로드할지"만 짧게 명시한다.
>
> 이 양식은 비어 있는 출발점이다. 폴더 구조·모드·메타블록 필드·
> 응답 스타일 등 도메인 특화 항목은 사용자와의 대화로 채워나간다.
> 채워야 할 항목은 `README.md`의 체크리스트를 따른다.

## 0. 이 볼트는 무엇인가

이 볼트는 단순 노트 저장소가 아니라 **AI와 공유하는 학습 데이터베이스**다.
사용자가 자기 도메인의 구조·규칙을 명시해두면, 그 볼트는 동시에:

- 사용자의 학습 자료 (개념·기록·요약)
- AI의 메모리 (어떤 자료가 어디에 어떻게 정리됐는지)

로 작동한다. 한쪽만 잘 만들어도 다른 한쪽에서 효과가 따라온다.

옵시디언은 이 공유 메모리를 받쳐주는 도구다. AI는 다음 기능을 적극 활용해
사용자의 학습 흐름을 만든다.

- **위키링크 + 자동 백링크**: `[[...]]` 한 방향만 적어도 옵시디언이 양방향
  그물을 자동으로 그려준다. "어느 단위에서 이 개념이 호명됐는가"를 검색
  없이 백링크 패널로 즉시 본다.
- **Dataview 인라인 필드**: `[필드:: 값]` 형태의 메타블록을 두면 출제·오답
  같은 누적치가 자동 집계되고, 회차별·과목별 약점이 한 줄로 추적된다.
- **그래프 뷰**: 노트 간 연결을 시각화. 학습 분포·약한 영역이 한눈에 보인다.
- **호버 미리보기**: 위키링크에 마우스만 올리면 연결 노트가 떠올라 흐름이
  끊기지 않는다.

도메인을 빌드할 때 이 도구 특성을 살리는 방향으로 설계한다. 예: 헤딩
이름은 위키링크 안정성을 위해 간결하게, 메타블록은 인라인 필드로,
회차·강의 같은 단위는 별도 노트로 분리해 백링크가 살아나게.

## 1. 대량 편집 가드레일

옵시디언 볼트는 위키링크·메타블록·반복 헤딩 등 동일 패턴이 많아 대량
치환 작업이 잦다. 동일 파일을 단순 Edit으로 반복하면 프롬프트 캐시가
매번 무효화되어 토큰이 수십 배로 불어난다.

| 트리거 | 전환 |
|---|---|
| 동일 파일 Edit 5회 초과 | Write 전체 재작성 검토 |
| 동일 패턴 치환 10회 초과 | `Edit replace_all` / `MultiEdit` / `Bash + sed` |
| 세션 누적 Edit 50회 초과 | 즉시 중단, 일괄 처리 재설계 |

예외: 패턴이 파일마다 다르거나, 매번 맥락 판단이 필요한 수정은 Edit
유지. 위 기준은 "같은 패턴을 기계적으로 반복할 때" 적용한다.

## 2. 운영 주의사항

- **단위 종료 시 정리는 메인 세션이 직접 처리한다.** 회차·단원·강의 등
  학습 단위가 끝난 뒤의 노트 정리 작업을 sub-agent에 자동 위임하지
  않는다. 트리거 일관성·시작·종료 오버헤드 문제가 생긴다.
- **파일 생성·수정·삭제 전 사용자에게 확인을 받는다.** 단, 정해진 정리
  절차(예: 회차 종료 정리) 안에서의 노트 갱신은 예외로 한다.
- **위키링크 일괄 치환은 `sed`/`perl -pi` 한 번으로 끝낸다.** 단순 Edit
  반복으로 풀지 않는다. (1번 가드레일 참고)
- **긴 PDF·자료를 참조해 채점·해설할 때는 답지·정답까지 모두 읽도록
  명시한다.** 자료가 길다고 Claude가 직접 풀어 답하면 토큰만 낭비된다.

## 3. 운영 설계 팁

- **규칙 분리** — 세부 규칙은 별도 파일로 두고 이 CLAUDE.md엔 로드
  시점만 적는다. 매 대화마다 모든 규칙을 컨텍스트에 올리지 않으니 토큰
  절약이 크다. 예: "복기 정리 트리거 → `규칙/세션복기.md` 로드".
- **모델 분리** — 메인 학습·정리는 Opus, 가벼운 출제·채점은 Sonnet.
  모드별 규칙 파일에서 권장 모델을 명시해두면 Claude가 시작 시 모델
  변경을 먼저 확인한다.
- **헤딩에 회차·번호 두지 않기** — 회차·강의·일자 같은 맥락은 헤딩이
  아니라 메타블록 인라인 필드(`이력`)에만 둔다. 헤딩은 개념 자체로
  유지해야 위키링크가 깨지지 않는다.
- **보조 섹션 막기** — Claude는 정리할 때 "답안 작성 팁", "키워드 매칭"
  같은 보조 섹션을 자꾸 만들려는 경향이 있다. 본문 비대화를 막으려면
  규칙으로 명시적으로 금지해두는 것이 좋다.
- **메타블록은 인라인 필드로** — 헤딩 옆 `#태그` 대신 Dataview 인라인
  필드(`[필드:: 값]`)를 쓰면 자동 집계·역방향 조회가 편하다.

## 4. 메타블록 기본 양식

각 개념 헤딩 바로 아래에 다음과 같이 둘 수 있다. 도메인에 맞춰 필드를
추가·삭제·이름 변경할 수 있다.

```markdown
## 개념명
- [출제:: 3] [오답:: 1] [헷갈림:: 0] [최근:: 2024-3회]
- 이력
  - [[2024-3회]] 5번 ✓
  - [[2024-1회]] 7번 ✗

(본문)
```

필드 의미(예시):

- `출제` — 누적 출제·등장 횟수
- `오답` — 누적 오답 횟수
- `헷갈림` — 현재 혼동 중이면 `1`, 정리되면 제거 또는 `0`
- `최근` — 가장 최근 등장 회차/강의 문자열
- `이력` — 위키링크 목록. `[[단위]] 위치 ✓/✗` 형식 권장

## 5. 그 외

이 외의 운영 규칙(폴더 구조·모드 이름·H2 네이밍·위키링크 연결 레벨·
정제 원칙·응답 스타일·모델 분리 정책·자료 위치 등)은 `README.md`의
체크리스트를 따라 사용자와 함께 정한다.
